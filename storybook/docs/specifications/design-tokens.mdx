# Design Tokens & Theming System

Cette page récapitule le fonctionnement et l'usage des Design Tokens du projet.<br/>
Notre architecture repose sur une source de vérité centralisée (Figma) et une génération automatisée.

## 1. Cycle de vie des Tokens (Workflow)

Il est essentiel de comprendre le flux de nos variables pour maintenir la cohérence du Design System :

* **Source de vérité (Figma & JSON)** : Les variables sont extraites de fichiers JSON situés dans `src/design-tokens/`.
* **Génération automatisée** : Un script `scripts/tokens.build.mjs` compile ces données en plusieurs formats cibles.
* **Fichiers SCSS/CSS (`.scss`)** : Générés dans `src/styles/src/generated/`, ce sont les variables (ex: `var(--...)`) que vous devez utiliser lorsque vous écrivez du style pour vos éléments de layout / composants.

## 2. Familles de Tokens

Les tokens sont divisés en plusieurs fichiers dans `src/styles/src/generated/` :

- **`_tokens-semantics.scss`** : Couleurs liées à une fonction d'interface (ex: action, feedback, navigation).
- **`_tokens-metrics.scss`** : Unités de mesure structurelles (espacement, arrondi, bordure).
- **`_tokens-typography.scss`** : Styles de texte (familles, graisses, tailles).
- **`_tokens-breakpoints.scss`** : Définition des points de rupture responsive.

## 3. Tokens Sémantiques (Couleurs)

Utilisez ces tokens pour garantir la cohérence visuelle et le support automatique du **Dark Mode**.

### Structure des Noms (Anatomie)

La nomenclature suit une logique de construction stricte : `[domaine]-[famille]-[niveau]-[propriété]-[état]`.
*(Note : le domaine `global` n'utilise pas de famille, sa structure est `global-[niveau]-[propriété]-[état]`)*.

## 5. Styles composites (`styles.json`)

Certains tokens Figma sont **composites** (plusieurs propriétés en un style : ombres…). Ils
vivent dans `src/design-tokens/styles.json` et sont générés différemment : le pipeline
**compose** une valeur CSS prête à l'emploi et l'expose en **double** — variable + classe
utilitaire — pour un usage aussi bien en SCSS qu'en HTML.

- Sortie : `--shadow-{catégorie}-{taille}` (valeur `box-shadow`) + `.shadow-{catégorie}-{taille}`.
- Config : output `format: "composite-styles"` dans `tokens.config.json`. Ajouter un type
  composite = étendre la table `COMPOSERS` dans `scripts/tokens.build.mjs`.
- Détail et aperçu : **Foundations → Shadows**.

## 6. Exemples d'Implémentation

```scss
// ✅ BON USAGE
.mon-composant {
  background-color: var(--global-high-surface-default);
  padding: var(--units-md);
  border-radius: var(--radius-default);
}

// ❌ À ÉVITER
.mon-composant {
  background-color: white; // Ne gère pas le Dark Mode
  padding: 12px;           // Valeur en dur interdite
}
```

## 7. Constantes structurelles partagées (`ui-config`)

En complément des tokens (thémables, runtime), les composants `ui-*` partagent des
**constantes structurelles** SASS (focus ring, bordure, taille des contrôles, transition) :
le *choix* du token pour un rôle donné, mutualisé et surchargeable à 3 niveaux
(global UI → catégorie → composant).

> Référentiel complet des variables, valeurs par défaut et mécanismes de surcharge :
> **Spécifications → Configuration (ui-config)**.
